## Context

当前战斗系统（`BattleSystem` 类）自项目创建以来逐步演进，添加了技能系统、动画效果等功能。代码中存在以下问题：

1. **技能系统重复**：`heroAttack()` 和 `useBattleSkill()` 两个方法处理相似的逻辑，但实现不同
2. **动画逻辑分散**：伤害数字、攻击特效、技能特效等动画逻辑分散在 `battle.js` 页面中
3. **状态混合**：战斗系统状态与页面 UI 状态耦合，难以独立测试和维护
4. **错误处理缺失**：缺少统一的错误捕获和用户反馈机制
5. **性能问题**：特效循环使用 `setTimeout` 链式调用，可能存在内存泄漏风险

## Goals / Non-Goals

**Goals:**
- 统一技能使用逻辑，消除代码重复
- 创建独立的动画管理模块，提高可维护性
- 分离战斗逻辑与 UI 状态，提高可测试性
- 添加统一的错误处理机制
- 优化渲染性能，减少不必要的重绘
- 提供战斗数据统计功能

**Non-Goals:**
- 重构整个游戏架构
- 修改战斗核心机制（伤害计算、闪避等）
- 更新 UI 布局和样式
- 添加新的战斗机制（如连击、合击等）

## Decisions

### 1. 统一技能入口
**决策**：创建 `executeSkill()` 方法作为统一入口，内部根据技能类型分发。

**理由**：保持现有 API 兼容性，`heroAttack()` 和 `useBattleSkill()` 作为适配器调用新方法。

**替代方案**：
- 完全删除旧方法 → 破坏性变更，需要更新所有调用点
- 保持两套逻辑 → 代码重复，维护成本高

### 2. 独立动画管理器
**决策**：创建 `BattleAnimationManager` 类管理所有战斗动画。

**理由**：
- 集中动画逻辑，便于调试和优化
- 支持动画队列和优先级
- 可独立于战斗系统进行单元测试

**替代方案**：
- 继续在页面中管理动画 → 难以测试和复用
- 使用动画库 → 引入外部依赖，增加包体积

### 3. 状态分离模式
**决策**：使用观察者模式，`BattleSystem` 只管理逻辑，通过事件通知 UI 更新。

**理由**：
- 解耦战斗逻辑和 UI 展示
- 便于未来支持多端渲染

**替代方案**：
- 保持现有混合状态 → 紧耦合，难以测试
- 使用状态管理库 → 引入复杂度，过度工程

### 4. 性能优化策略
**决策**：使用 `requestAnimationFrame` 替代 `setTimeout`，实现脏标记渲染。

**理由**：
- 与浏览器渲染周期同步，提高流畅度
- 脏标记避免不必要的重绘

**替代方案**：
- 保持 `setTimeout` → 简单但不高效
- 定时重绘 → 浪费性能

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 重构可能引入新 Bug | 充分测试现有功能，保持 API 兼容 |
| 动画逻辑迁移可能影响性能 | 逐步迁移，性能对比测试 |
| 状态分离可能导致事件管理复杂 | 使用统一的事件命名规范 |
| 统计功能增加内存占用 | 提供统计开关，仅在需要时记录 |

### Trade-offs
- **代码清晰度 vs 引入抽象**：引入新的管理类增加抽象层次，但提高可维护性
- **性能 vs 开发复杂度**：脏标记渲染需要额外状态管理，但显著提升性能
- **兼容性 vs 简洁性**：保留旧方法确保兼容，但增加代码行数

## Migration Plan

1. **阶段一**：创建新模块（动画管理器、统计模块）
2. **阶段二**：重构 `BattleSystem`，添加统一技能入口
3. **阶段三**：迁移动画逻辑到动画管理器
4. **阶段四**：分离状态管理，建立事件通知机制
5. **阶段五**：添加错误处理和统计功能
6. **阶段六**：全面测试和性能验证

**回滚策略**：每个阶段提交独立 Git 提交，便于回滚到任意阶段。
