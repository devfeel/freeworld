## Context

当前战斗界面由简单的 UI 面板和小型 Canvas（300rpx 高）组成，缺乏沉浸感。玩家反馈战斗过程信息不足，状态变化（如 Buff/Debuff）没有可视化反馈。需要在不牺牲性能的前提下提升视觉体验。

## Goals / Non-Goals

**Goals:**
- 提升战斗场景的视觉冲击力和沉浸感
- 清晰展示状态效果，让玩家了解当前战斗状态
- 增强战斗日志的信息传递效率和可读性
- 添加粒子特效和屏幕震动，增强动作反馈
- 在低端设备上保持流畅运行（平衡性能和耗电）

**Non-Goals:**
- 修改战斗核心逻辑（伤害计算、回合顺序等）
- 完全重写游戏渲染引擎
- 添加复杂的 3D 效果或 WebGL 渲染
- 改变战斗机制（如添加连击系统）

## Decisions

### 1. 战斗场景扩展策略
**决策**：Canvas 高度扩展至 400rpx，使用分层绘制（背景层 → 角色层 → 特效层 → UI 层）。

**理由**：
- 保持与现有 Canvas 绘制逻辑兼容
- 分层绘制便于优化（不需要每一帧都重绘背景）
- 400rpx 在主流设备上可完整显示

**替代方案**：
- 使用多 Canvas 分层 → 增加复杂度，同步问题
- 使用 WXML 覆盖层 → 难以实现角色动画

### 2. 背景装饰实现
**决策**：使用程序化绘制而非图片资源，绘制石柱、地面纹理、氛围光效。

**理由**：
- 无需加载外部图片，减少包体积
- 支持动态效果（火焰闪烁、雾气流动）
- 与像素风格一致

**替代方案**：
- 使用背景图片 → 增加 HTTP 请求，缓存复杂

### 3. 粒子系统实现
**决策**：创建 `ParticleSystem` 类，使用对象池复用粒子，支持多种粒子类型（伤害、暴击、火花）。

**理由**：
- 对象池避免频繁 GC，提升性能
- 统一管理所有粒子效果
- 支持配置化的粒子属性（速度、颜色、寿命）

**替代方案**：
- 每次创建新粒子 → GC 压力大，性能差
- 使用动画库 → 增加包体积，过度工程

### 4. 屏幕震动实现
**决策**：使用 CSS transform 动画而非 Canvas 绘制，通过添加/移除 class 控制。

**理由**：
- 利用浏览器/系统原生动画，性能更好
- 与 Canvas 渲染解耦
- 易于控制震动强度和时长

**替代方案**：
- Canvas 偏移绘制 → 需要每一帧计算偏移量
- 模拟震动（角色抖动）→ 反馈不够强烈

### 5. 性能平衡策略
**决策**：三档性能配置（高性能/平衡/低功耗），根据设备信息自动选择，并提供手动切换。

**理由**：
- 低端设备自动使用低功耗模式
- 高端设备可享受完整视觉效果
- 玩家可手动调整偏好

**配置对比**：
| 档位 | 粒子数量 | 屏幕震动 | 背景效果 |
|------|-----------|-----------|-----------|
| 高性能 | 100+ | 强 | 完整 |
| 平衡 | 50 | 中 | 部分 |
| 低功耗 | 10 | 弱 | 静态 |

### 6. 状态效果展示位置
**决策**：在技能槽下方独立区域显示，支持最多 6 个状态图标。

**理由**：
- 不遮挡战斗画面
- 与技能槽视觉关联（技能往往带状态效果）
- 固定位置便于玩家观察

**替代方案**：
- 悬浮在角色上方 → 可能与伤害数字冲突
- 替换技能图标 → 影响技能选择

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| Canvas 扩大可能影响低端设备性能 | 添加性能检测，动态调整粒子数量 |
| 粒子系统可能导致内存泄漏 | 严格的粒子回收机制，最大粒子数限制 |
| 屏幕震动影响可读性 | 震动时长控制在 200ms 内 |
| 状态图标过多导致拥挤 | 限制最多显示 6 个，优先显示重要状态 |

### Trade-offs
- **视觉丰富度 vs 性能**：更多效果更震撼，但需要更高性能
- **信息密度 vs 清晰度**：状态效果提供更多战斗信息，但可能分散注意力
- **自动性能检测 vs 用户控制**：自动选择体验好，但用户可能想手动调整

## Migration Plan

1. **阶段一**：扩展 Canvas 高度，调整布局
2. **阶段二**：添加背景装饰绘制
3. **阶段三**：实现粒子系统基础功能
4. **阶段四**：实现屏幕震动效果
5. **阶段五**：添加状态效果展示
6. **阶段六**：优化战斗日志样式
7. **阶段七**：性能检测和配置系统
8. **阶段八**：测试和调优

**回滚策略**：每个阶段独立提交，可通过配置开关禁用新功能。

## Open Questions

- 背景装饰是否需要根据地下城类型变化（如火焰地牢、冰雪洞穴）？
- 状态效果是否需要动画（如火焰图标燃烧动画）？
- 屏幕震动是否需要支持不同类型（震动、闪烁、色差）？
- 低功耗模式是否需要持久化到本地存储？
